# Trinity AI Labs — Brand Voice Guide

## Core Identity

Trinity is a **next-generation IDE** — not a copilot, not a code generator, not an assistant. It plans, executes, reviews, and ships software autonomously through a multi-agent pipeline. The voice reflects this: confident, forward-looking, and action-oriented.

## Tone Characteristics

### Direct & Confident
Write with conviction. Don't hedge. Don't qualify.

- **Yes**: "Trinity ships entire features while you sleep."
- **No**: "Trinity can potentially help you ship features more efficiently."

### Forward-Thinking & Visionary
Position Trinity as defining a new category, not iterating on an old one.

- **Yes**: "Your IDE should understand your codebase. Not just the file you have open."
- **No**: "Trinity is an improved code editor with AI capabilities."

### Technical but Accessible
Explain complex concepts without unnecessary jargon. When using technical terms, show what they mean in practice.

- **Yes**: "Each story runs in its own git worktree — complete isolation, zero merge conflicts."
- **No**: "The system utilizes containerized execution environments with git-based workspace isolation mechanisms."

### Slightly Irreverent
Willing to challenge conventions. Not snarky, but not afraid to call things out.

- **Yes**: "Your 'IDE' is just a text editor with a terminal bolted on."
- **No**: "Traditional IDEs may have some limitations."

### Action-Oriented
Emphasize outcomes and results, not just features.

- **Yes**: "Describe it. Ship it."
- **No**: "Trinity provides AI-powered development capabilities."

## Vocabulary

### Power Verbs (use these)
ship, build, execute, orchestrate, plan, review, learn, discover, compound, isolate, coordinate, analyze, audit, refactor

### Results Words
autonomous, integrated, structured, parallel, real (as in "real execution data"), compound (as in "compound learning")

### Avoid
leverage, utilize, empower, solution, cutting-edge, next-level, game-changing, state-of-the-art, seamlessly, robust

## Sentence Structure

- **Headlines**: Short and punchy. Periods for emphasis. "Describe it. Ship it." not "Describe it and ship it!"
- **Body copy**: Mix of short (1 sentence) and medium (2-3 sentence) paragraphs
- **Lists**: Use em dashes or parallel structure. "Planning. Execution. Review. Knowledge."
- **Active voice**: "Agents write PRs" not "PRs are written by agents"

## Product Framing

### The Pipeline Metaphor
Trinity's multi-agent pipeline is the central metaphor. Reference it naturally:
- Story pipeline: Analyst → Implementer → Auditor (simplify loop) → Documenter
- Checkpoint/release pipeline: Analyst → Audit/Fix loop → Refactor → Audit/Fix loop → Documenter
- Planning: Architect → Story Writer → Dependency Mapper → Calibrator

### Specificity Over Generality
Don't say "it helps with planning." Say "4-phase planning: Architect → Story Writer → Dependency Mapper → Calibrator."

### Problem-First Narrative
Start with what's broken before showing the fix. The reader should feel the pain before seeing the solution.

## Writing for Different Audiences

### Developers (primary)
Speak their language. Reference real tools (git, CI, PRs). Use code examples. Show the terminal output. They'll respect specificity and distrust vague claims.

### Technical Leaders / CTOs
Focus on outcomes: shipping speed, quality metrics, team efficiency. Show how Trinity fits into their workflow without replacing their team's judgment.

### General Tech Audience
Lead with the vision. Use analogies. Compare to what they already know ("like having a senior dev who never sleeps"). Less jargon, more outcomes.
